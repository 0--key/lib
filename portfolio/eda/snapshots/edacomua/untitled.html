Начнём построение структурной схемы с логического начала - структуры обмена информацией. Но для этого рассмотрим весь объём подлежащих обработке данных.
    Данные, содержащиеся в базе данных по товарам для удобства манипулирования разобьём на две части, первую - глобальные данные едины для всей базы данных), и вторую - локальные данные (свойственные тому или иному объекту розничной реализации). Дадим им имена

 global_meal:
create database meal_store;
connect meal_store;
use meal_store;
create table meal_global (id bigint(13) unsigned, meal_name varchar(25), manufacturer varchar(15), meal_type varchar(15),  features blob, weight_brutto_kg decimal(2,2) unsigned, size_L decimal(2,2) unsigned, author varchar(8), time timestamp, photo_id smallint(4) unsigned, primary key(meal_type));
  • ID (13) - 13 цифр штрих-кода;
  • meal_name - название продукта;
  • manufacturer - производитель;
  • meal_type - тип группы;
  • features - описание;
  • weight_brutto_kg - вес вместе с упаковкой в килограммах;
  • size_L - объём вместе с упаковкой в литрах;
  • author - автор;
  • time - время занесения записи
  • photo_id - индекс на фотографию.
    Основным ключём сделано поле meal_type, т.к. чаще всего запросы будут именно по типу группы (молочные продукты, алкоголь, товары для детей и т.п.).  
Основные свойства таблицы:
  1. Доступна для чтения и записи только пользователям из таблицы runners с типом user;
  2. Доступна только для чтения пользователям из таблицы customers со статусом normal;
    и local_meal:

create table meal_local (id bigint(13) unsigned, price_UH decimal(4,2) unsigned, town varchar(10), store_name varchar(10), allocation varchar(4), presence enum('yes','no'), author varchar(8), primary key(id));
  • ID - 13 цифр штрих-кода;
  • price - цена;
  • town {Zap, Ene, ...};
  • store_name - имя объекта розничной реализации (адрес);
  • allocation {A79v} - расположение товара на торговой площади;
  • presence {yes, no} - наличие;
  • author - автор записи;
  • time - время занесения записи.
    Её свойства:
  1. Доступна для чтения пользователям из таблицы customers;
  2. Доступна для записи пользователям из таблицы runners со статусом user и только, если autor;
    Данные, содержащиеся в базе данных клиентов, назовём её

customers:

create table customers(name varchar(25), nick varchar(8), password varchar(6), cookie varchar(8), town varchar(25), allocation varchar(4), streettype enum('улица', 'проспект', 'бульвар', 'переулок'), streetname varchar(15), build_no smallint(3) unsigned, korpus tinyint(1) unsigned, flat_no smallint(3) unsigned, access_mode enum('свободный','домофон','консъерж','код'), door_code smallint(4) unsigned, phone_no int(9) unsigned, priority enum('first','high','low','deny'));
  • cu_name;
  • cu_nick;
  • cu_password;
  • cookie;
  • cu_address;
  • cu_phone;
  • cu_priority {deny, low, normal, high};
    Её свойства:
  1. Доступна для записи пользователю customer;
  2. Доступна для чтения пользователю machine_head;
  3. Поле priority доступно для записи пользователю machine_head
    И теперь последнее звено - курьеры,

runners:

create table runners(name varchar(15), nick varchar(8), password varchar(6), phone_no int(9) unsigned, photo_index smallint(4) unsigned, scan_index smallint(4) unsigned, status enum('busy','ready','offline'), type enum('ordinary','user','deny'));
  • ru_name;
  • nick;
  • password;
  • ru_phone;
  • ru_photo_index;
  • ru_scan_index;
  • ru_status{ready, busy, offline};
  • type{ordinary, user, deny}.
    Свойства этой таблицы:
  1. Запись и чтение разрешены пользователю machine_head;


На этом перечень таблиц, достаточных для нормального функционирования системы окончен. Следующие таблицы носят вторичный характер и предназначены для удобства обслуживания базы данных.
    Для регистрации произведённых заказов создадим таблицу orders:

create table orders (num smallint(5) unsigned not null auto_increment, nick varchar(8), runner_nick varchar(8), status enum('on','off','deny'), postscript tinyblob, time timestamp, primary key (num));

    Её свойства:
  1. Запись осуществляет machine_head;
  2. Чтение может осуществлять supervisor.

    А для хранения информации о составе заказа создадим таблицу inorder:

create table inorder (num smallint(5) unsigned, id bigint(13) unsigned, psc tinyint(3) unsigned, primary key (num));

    Её свойства:
  1. Запись осуществляет machine_head;
  2. Чтение может осуществлять supervisor.
